## Clarifications

### Session 2026-06-08

- Q: How are ingested documents scoped, and what is the dedup key? → A: **Client-scoped** documents with dedup key `(client_id, source, external_id)`; within one client a paper surfaced by several of that client's watchlists is stored **once**, with the surfacing watchlist(s)/run recorded as provenance via a separate link. Cross-client dedup is intentionally **not** done in this spec (each client keeps its own copy for strict tenant isolation); a `client_documents` junction table that stores a shared paper once across clients is a documented future improvement.
- Q: How does the manual trigger execute the (slow, multi-source) ingestion work? → A: **Background task, in-process** — the trigger endpoint creates the run record, starts the ingestion in the background, and returns immediately with the run ID; the caller polls/reads the run record for status and counts. This avoids request timeouts and keeps the full ARQ queue + cron orchestration (dead-letter, idempotency, SLA) in **spec 11**; the ingestion logic is written as a reusable function that spec 11 will enqueue to the ARQ worker.
- Q: Where does MeSH validation get its controlled-vocabulary "dictionary," and how is synonym expansion handled across sources? → A: Validate against a **bundled slim MeSH term list** (term → canonical form, shipped with the app — offline, deterministic, CI-testable; refreshed ~yearly), and delegate **synonym expansion to PubMed's native server-side MeSH expansion** (no expansion engine built here). MeSH applies only to the **MeSH-aware** sources: **PubMed** (full) and **Europe PMC** (partial); **openFDA and the regulatory feeds (MedWatch/EMA/MHRA) are queried by drug name + keywords, with no automatic synonym expansion**. Each adapter consumes only the watchlist fields it understands. Cross-source automatic drug-synonym coverage (RxNorm/drug disambiguation) is a documented future improvement, out of scope here.
- Q: Which source adapters must be fully working for this spec to be "done"? → A: **All six are required** — PubMed, Europe PMC, openFDA (FAERS + drug labels), FDA MedWatch, EMA, and MHRA — matching the build plan. To manage schedule risk, the two flakiest feeds (**EMA, MHRA**) are **sequenced last** as a buffer: the spec is demonstrably complete with the first four while EMA/MHRA are finished, but all six are in scope for done.
- Q: When are a watchlist's MeSH terms validated? → A: **At save time** — adding/editing a MeSH term (enhancing the spec-3 watchlist write path, which deferred this check to spec 4) validates it against the bundled list and stores a validity flag on the item, so the term's validity is **visible whenever the watchlist is viewed**; ingestion **re-checks defensively** at run time. Validation **flags, never blocks** (an invalid term is marked for correction, the admin's term is never silently dropped or rejected).

### Session 2026-06-08 (pass 2)

- Q: Does the same paper from two sources collapse into one document? → A: **Yes — dedup across sources** by a **normalized external identifier**, key `(client_id, normalized_external_id)`. One real paper = one document even when it appears in several sources (e.g., PubMed + Europe PMC); the document records **all contributing sources** and its source-reliability tier is the **highest** among them. Identifier precedence for normalization: **DOI → PubMed ID → source-native alert/record ID**. This keeps corroboration honest (one paper never counts as two sources) and avoids double-embedding downstream.
- Q: How much literature does a run pull — everything, or just what's new? → A: **Incremental** — each run fetches only records newer than that watchlist's **last successful run watermark** (tracked per `(watchlist, source)`, advanced only when that source succeeds, so a failed source re-tries its window next run). The **first run** uses a bounded, configurable **initial lookback window** (default ~12 months), and every source query is bounded by a **per-source result cap** as a safety valve. Dedup still guards against any overlap.

## User Scenarios & Testing *(mandatory)*

Pantera monitors published medical literature and regulatory safety communications for adverse drug events on behalf of its business clients. Spec 3 gave each client first-class watchlists describing **what** to monitor (drugs, MeSH terms, keywords) and **under what limits** (cadence, severity threshold, cost budget). This feature is the first spec that reaches **outside the platform**: given a watchlist, it queries external sources — PubMed (with MeSH expansion), Europe PMC, openFDA (FAERS + drug labels), and the regulatory safety feeds FDA MedWatch, EMA, and MHRA — and brings their results in as **stored documents**, each normalized to a common shape, tagged with a **source-reliability** tier, and **deduplicated by external identifier** so the same record is never stored twice. It also takes ownership of **MeSH validation** (deferred from spec 3): resolving each watchlist's MeSH terms against the controlled vocabulary so PubMed targeting is precise.

This spec **does not** parse documents into chunks, embed them, classify them, or draft anything — it only fetches and durably persists raw + normalized source records so the later specs (parsing/embedding = spec 6, retrieval = spec 7, triage = spec 8, drafting = spec 9) have a corpus to act on. Ingestion is started by a **manual trigger** in this spec; the scheduled/cron cadence that automates it is spec 11. This spec is **backend/API only**; the Admin Console "manual trigger" button is a later frontend slice.

### User Story 1 - Manually trigger an ingestion run and persist new documents (Priority: P1)

An admin (acting within their own client) manually triggers an ingestion run for one of their client's active watchlists. The platform queries the primary literature source for records matching the watchlist's drugs/terms, normalizes each result into a common document shape, and persists the new documents scoped to that client — recording what source each came from, its external identifier, and when it was fetched. This is the irreducible "pull literature in and store it" capability that every later spec depends on.

**Why this priority**: Without the ability to trigger a fetch and durably store the results, there is no corpus — nothing for parsing, retrieval, triage, or drafting to consume. A single working source end-to-end (trigger → fetch → normalize → persist, client-scoped) is the minimum viable, independently demonstrable slice of this spec.

**Independent Test**: As an admin, trigger an ingestion run for one of your client's active watchlists; confirm new documents are persisted, scoped to your client, each carrying its source, external identifier, normalized metadata, and fetched-at timestamp; confirm an admin of another client cannot trigger or see this client's ingestion. Confirm triggering against a deactivated/empty/other-client watchlist is refused with a clear message.

**Acceptance Scenarios**:

1. **Given** an active watchlist with at least one monitored item, **When** an admin triggers an ingestion run for it, **Then** matching external records are fetched, normalized, and persisted as documents scoped to that watchlist's client, each with its source, external identifier, metadata, and fetched-at time, and the run reports how many documents were created.
2. **Given** an admin of client A, **When** they attempt to trigger ingestion for a watchlist belonging to client B, **Then** the request is refused and no client B data is fetched or revealed.
3. **Given** a deactivated (soft-deleted) or empty watchlist, **When** an admin attempts to trigger ingestion for it, **Then** the request is refused with a clear validation message and no documents are created.
4. **Given** a reviewer (non-admin), **When** they attempt to trigger an ingestion run, **Then** the request is refused as forbidden, but they may view their client's ingested documents and run results.
5. **Given** an external source returns no matching records, **When** the run completes, **Then** it completes successfully reporting zero new documents (an empty result is not an error).

### User Story 2 - Ingest from all configured sources with source-reliability tagging (Priority: P1)

An ingestion run fans out across **all** of the platform's configured external sources — PubMed (with MeSH expansion), Europe PMC, openFDA (FAERS adverse-event reports and drug labels), and the regulatory safety feeds FDA MedWatch, EMA, and MHRA — each behind a uniform adapter that normalizes its native format into the common document shape. Every stored document is tagged with a **source-reliability** tier that reflects how authoritative its origin is (regulatory alert > peer-reviewed > preprint > case report), so later triage and corroboration can weight evidence by trust.

**Why this priority**: Pantera's value is breadth across heterogeneous, multi-format sources — a single source misses signals and the regulatory feeds are the highest-confidence adverse-event origins. The source-reliability tag is required at ingestion time because it is an intrinsic property of where a document came from and is consumed by every downstream spec. This ships alongside US1 as the other half of the minimal "real, multi-source corpus" MVP.

**Independent Test**: Trigger a run for a watchlist and confirm each configured source is queried and its results normalized into the common shape; confirm each persisted document carries the correct reliability tier for its origin (e.g., an FDA MedWatch alert is tagged `regulatory_alert`, a PubMed article `peer_reviewed`, a preprint `preprint`, an openFDA FAERS report `case_report`); confirm a record from one source is distinguishable from an identical-topic record from another source by its source field.

**Acceptance Scenarios**:

1. **Given** an active watchlist, **When** an ingestion run executes, **Then** every configured source is queried using the watchlist's drugs/MeSH/keywords and each returned record is normalized into the common document shape (source, external identifier, title/summary, publication or alert date, origin URL, and the raw payload) regardless of the source's native format.
2. **Given** documents fetched from different source types, **When** they are persisted, **Then** each is tagged with the correct source-reliability tier from the fixed set `{regulatory_alert, peer_reviewed, preprint, case_report}`.
3. **Given** a regulatory safety feed (FDA MedWatch / EMA / MHRA), **When** its alerts are ingested, **Then** they are tagged `regulatory_alert` (the highest-confidence tier).
4. **Given** the same drug appears in **different** records across multiple sources, **When** a run completes, **Then** the corpus contains a distinct document per distinct record, each carrying its contributing source(s) and reliability tier (whereas the *same* record appearing in multiple sources is collapsed into one document — see User Story 3).

### User Story 3 - Deduplicate ingested documents by external identifier (Priority: P2)

When the same record is encountered more than once — across repeated runs of the same watchlist, or because overlapping sources (e.g., PubMed and Europe PMC) surface the same publication — the platform recognizes it by its **normalized external identifier** (DOI → PubMed ID → source-native alert/record ID) and does not store a redundant copy. A re-run brings in only genuinely new records; an already-stored record is reused — the new source is added to its contributing sources and its reliability tier raised to the highest contributing tier — never duplicated.

**Why this priority**: Without dedup, every re-run inflates the corpus with copies, wasting later parsing/embedding/triage work and distorting corroboration counts (the same paper must never be counted as two independent sources). It is a guardrail layered on the ingestion of US1/US2, so it is P2 — the corpus is demonstrable without it, but it is required before scheduled re-runs (spec 11) are safe.

**Independent Test**: Trigger an ingestion run twice for the same watchlist with no new external records in between; confirm the second run creates zero new documents and reports the duplicates as skipped; confirm that when two configured sources return the same record (same external identifier), it is stored once; confirm dedup is scoped per client (the same external record may exist independently for two different clients that both monitor it).

**Acceptance Scenarios**:

1. **Given** a document already stored for a watchlist's client, **When** a later run encounters the same normalized identifier (from the same or a different source), **Then** no duplicate document is created and the run reports it as a skipped duplicate.
2. **Given** two configured sources that both surface a record with the same normalized identifier within one run, **When** the run persists results, **Then** the record is stored once (not once per source) with both sources recorded as contributing and the reliability tier set to the highest contributing tier.
3. **Given** the same external record is relevant to two different clients, **When** both clients ingest it, **Then** each client has its own copy (dedup does not cross the tenant boundary), and neither can see the other's.
4. **Given** a re-run with a mix of previously-seen and brand-new records, **When** it completes, **Then** only the new records are created and the run accurately reports created-vs-skipped counts.

### User Story 4 - Validate and resolve watchlist MeSH terms (Priority: P2)

The platform validates each watchlist's MeSH terms against the controlled MeSH vocabulary (the validation deferred from spec 3, where MeSH terms were stored free-form). Valid terms are resolved/normalized for precise PubMed targeting (MeSH expansion, so a heading also catches its vocabulary-mapped synonyms); invalid or unrecognized terms are flagged back to the admin so they can be corrected, without silently dropping monitoring coverage.

**Why this priority**: MeSH validation makes PubMed retrieval precise and closes the explicit carryover from spec 3, but ingestion still functions on drugs and keywords without it, so it is P2 rather than part of the P1 MVP. It must land in this spec because this is where the PubMed/MeSH adapter lives.

**Independent Test**: Submit a watchlist whose MeSH terms include both valid headings and an invalid string; confirm valid terms are accepted and resolved against the controlled vocabulary, and invalid terms are reported as such with a clear message; confirm a PubMed ingestion run uses the validated/expanded MeSH terms for targeting; confirm validation does not mutate or remove the admin's stored terms without surfacing the result.

**Acceptance Scenarios**:

1. **Given** an admin adding or editing a MeSH term on a watchlist, **When** the term is saved, **Then** it is validated at save time against the bundled list — recognized terms are marked valid (and resolved to their canonical form) and unrecognized terms are saved but flagged invalid with a clear, actionable message (the term is never silently dropped or rejected).
2. **Given** validated MeSH terms, **When** a PubMed ingestion run executes, **Then** it targets PubMed using the resolved MeSH terms with vocabulary expansion, improving precision over a raw keyword query.
3. **Given** an invalid MeSH term is flagged, **When** an admin reviews the watchlist, **Then** the invalid term is visibly marked so it can be corrected, and the rest of the watchlist's monitoring is unaffected.
4. **Given** a watchlist with no MeSH terms (only drugs/keywords), **When** an ingestion run executes, **Then** it proceeds using the available drugs/keywords without error.

### User Story 5 - Resilient, observable ingestion runs (Priority: P3)

Each ingestion run is a tracked, audited unit of work that isolates per-source failures: if one source is unreachable or errors, the run still ingests from the others and records which sources succeeded, which failed, and the document counts (fetched / created / skipped-duplicate / errored). External calls are made resiliently (timeouts and bounded retries) so a transient outage does not corrupt the run, and the trigger is recorded in the audit log attributed to the acting admin.

**Why this priority**: Resilience and observability make ingestion safe to operate and to schedule later (spec 11), and an audit trail is a compliance requirement — but the core ingest-and-store value is demonstrable before this hardening, so it is P3, layered on once the runs themselves work.

**Independent Test**: Trigger a run while one source is made to fail; confirm the run completes, ingests from the healthy sources, and records the failed source as failed with its error noted while the others report success; confirm the run records accurate per-status counts; confirm the trigger produced exactly one audit-log entry attributed to the acting admin.

**Acceptance Scenarios**:

1. **Given** a multi-source run where one source errors or times out, **When** the run executes, **Then** the other sources are still ingested, the failed source is recorded as failed (with its error captured) and does not abort the whole run, and the run's overall status reflects partial success.
2. **Given** any ingestion run, **When** it completes, **Then** it records per-source outcomes and document counts (fetched, created, skipped-as-duplicate, errored) that are retrievable for that client.
3. **Given** a transient external failure, **When** the platform calls that source, **Then** the call is retried within bounded limits before being recorded as failed, and a permanent (non-retryable) error is not retried.
4. **Given** an admin triggers a run, **When** the run is started, **Then** exactly one audit-log entry is recorded attributing the trigger to that admin and naming the target watchlist.

### Edge Cases

- **Trigger against an ineligible watchlist**: Triggering ingestion for a deactivated, empty (no items), or non-existent watchlist, or one belonging to another client, is refused with a clear message and creates no documents.
- **Cross-tenant isolation**: An admin or reviewer of one client must never trigger ingestion for, or read the ingested documents/run results of, another client. Every stored document carries `client_id` and is filtered by it on every read.
- **Empty result set**: A source returning no matches is a successful zero-result outcome, not an error.
- **Duplicate within a single run**: The same normalized identifier surfaced twice in one run (e.g., by two sources) is stored once, with both sources recorded as contributing.
- **Duplicate across runs**: A previously-stored normalized identifier encountered in a later run is reused (its contributing sources/provenance updated, reliability tier raised if a higher-tier source surfaced it), never duplicated; corroboration must never see one paper as two sources.
- **Same record, two clients**: Dedup is scoped per client; the same external record may be stored once per client that monitors it, with no cross-tenant sharing in this spec.
- **Partial source failure**: One source failing (network error, timeout, rate-limit, malformed response) must not abort the run or lose the other sources' results; the failure is isolated and recorded.
- **Malformed / unparseable source record**: An individual record that cannot be normalized is skipped and recorded as errored without failing the whole source or run.
- **Invalid MeSH term**: An unrecognized MeSH term is flagged for correction rather than silently dropped or silently passed to PubMed as a literal.
- **Rate limiting / polite external use**: External sources must be queried within their usage limits (bounded concurrency, backoff) so Pantera is not throttled or blocked; this is an operational property of every adapter.
- **Concurrent / overlapping triggers**: Two ingestion runs for the same watchlist overlapping in time must not corrupt the corpus or create duplicates (dedup by normalized identifier remains authoritative, enforced at the storage layer).
- **Source credential missing/expired**: A source requiring an API key whose credential is absent is recorded as a failed source for the run (not a crash), and the rest proceed.
- **Watchlist deactivated mid-run / after trigger**: If the target watchlist is deactivated or its client suspended after a run starts, the in-flight run finishes and records its result, but no new run is accepted; already-ingested data is preserved, not deleted (FR-022).
- **Interrupted run**: If the process stops mid-run, the run must not stay `running` forever — it is reconciled to `failed`, and re-triggering is safe (no duplicates) thanks to dedup + watermarks (FR-024).
- **Dateless source / no usable record date**: A source exposing no usable date for windowing is paged by its native ordering/cursor with dedup suppressing already-seen records (FR-021).
- **Missing/stale MeSH artifact**: If the bundled MeSH list is absent or unreadable, MeSH terms are marked `unvalidated` and ingestion proceeds on drugs/keywords rather than crashing (FR-009).
- **Unidentifiable record**: A record with no DOI/PubMed ID/alert-or-record ID and no stable source key is recorded as errored, never stored (FR-006/FR-014).
- **Audit continuity**: Every trigger is audit-logged and attributed to the acting admin via the spec-2 human-actor mechanism.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST allow an `admin` to **manually trigger an ingestion run** for one of their own client's **active, non-empty** watchlists, and MUST refuse a trigger targeting a deactivated, empty, non-existent, or another client's watchlist with a clear validation message and no side effects.
- **FR-002**: System MUST, on a triggered run, query the platform's configured external sources using the target watchlist's drugs, MeSH terms, and keywords, and MUST persist each newly-discovered external record as a **document** scoped to that watchlist's client. Each source adapter MUST use only the watchlist fields it understands: MeSH-aware sources (PubMed; Europe PMC partially) use MeSH terms (plus drugs/keywords), while openFDA and the regulatory feeds (MedWatch/EMA/MHRA) are queried by drug name + keywords.
- **FR-003**: System MUST support ingestion from all of these source types behind a uniform adapter contract, and **all six are required for this spec to be complete**: **PubMed** (with MeSH expansion), **Europe PMC**, **openFDA** (FAERS adverse-event reports and drug labels), and the regulatory safety feeds **FDA MedWatch**, **EMA**, and **MHRA**. Adding or removing a source MUST NOT require changing the stored document shape. (To manage schedule risk, EMA and MHRA — the flakiest feeds — are sequenced last; the other four plus the adapter contract are delivered first, but EMA/MHRA remain in scope for done.)
- **FR-004**: System MUST normalize every ingested record, regardless of its native format, into a **common document shape**. The following fields are **REQUIRED** and a record missing any of them MUST be recorded as errored (FR-014) rather than stored: the **normalized external identifier** (the dedup key), at least one **contributing source**, the owning `client_id`, the **source-reliability** tier, the **fetched-at** timestamp, and the **raw source payload(s)** (retained for the later parsing spec). The following are **OPTIONAL/best-effort** (a source may not supply them): title/summary, publication or alert date, and origin reference (e.g., URL). The document is **client-scoped** (it carries `client_id`, not a single owning watchlist); the watchlist(s) and run that surfaced it MUST be recorded as **provenance** via a separate link, so a paper relevant to several of a client's watchlists is stored once while still showing which watchlists wanted it.
- **FR-005**: System MUST tag every ingested document with a **source-reliability** tier from the fixed **ordered** set `regulatory_alert` > `peer_reviewed` > `preprint` > `case_report` (left = most authoritative; this ordering is canonical and defines "highest"). The mapping MUST be fully enumerated: FDA MedWatch / EMA / MHRA feeds → `regulatory_alert`; PubMed/Europe PMC peer-reviewed articles and openFDA **drug labels** → `peer_reviewed`; PubMed/Europe PMC records flagged as **preprints** (e.g., bioRxiv/medRxiv) → `preprint`; openFDA **FAERS** individual adverse-event reports → `case_report`. When a document has more than one contributing source (see FR-006), its stored tier MUST be the **highest** in this ordering among them. The tier MUST be stored on the document for downstream consumption.
- **FR-006**: System MUST **deduplicate** ingested documents by a **normalized external identifier** using the key `(client_id, normalized_external_id)`, where the identifier is normalized by the precedence **DOI → PubMed ID → source-native alert/record ID**. Within a client's corpus a record whose normalized identifier is already stored MUST NOT create a duplicate — whether encountered again in a later run, surfaced twice within a single run, surfaced by **more than one source** (e.g., PubMed and Europe PMC; the contributing source is added to the existing document and its reliability tier raised to the highest contributing tier per FR-005), or surfaced by more than one of that client's watchlists (the new watchlist is added to the document's provenance). Dedup is scoped **per client** (the same external record MAY exist once per client that monitors it; cross-client dedup is out of scope — see Assumptions) and MUST guarantee that one real paper is stored exactly once per client so corroboration never counts it as two sources. When a record carries **none** of DOI / PubMed ID / source-native alert-or-record ID, the system MUST treat the source's own stable record key as the identifier; if the source provides no stable identifier at all, the record MUST be recorded as errored (FR-014) rather than stored, so an unidentifiable record can never silently duplicate.
- **FR-007**: System MUST scope every ingested document and every ingestion-run record to a single client; all trigger and read operations MUST be restricted to the acting user's own client, and no operation may trigger ingestion for, read, or enumerate another client's documents or runs. Cross-tenant access MUST be refused (per the constitution's multi-tenant isolation principle).
- **FR-008**: System MUST enforce that only an `admin` may trigger an ingestion run; a `reviewer` MAY view their own client's ingested documents and run results but MUST NOT trigger runs; unauthenticated callers MUST be refused before any tenant check.
- **FR-009**: System MUST **validate watchlist MeSH terms** against a **bundled slim MeSH term list** (term → canonical form, shipped with the application as a local data artifact; this is the validation deferred from spec 3): recognized terms MUST be marked valid and resolved to their canonical form, and unrecognized terms MUST be flagged invalid with a clear, actionable message, without silently dropping or mutating the admin's stored terms. Validation MUST function offline/deterministically (no per-validation external call). Validation MUST run **at watchlist save time** (when a MeSH term is added/edited — enhancing the spec-3 watchlist write path) and MUST **persist a validity state on each MeSH item** so it is visible whenever the watchlist is read; the ingestion run MUST **re-check defensively** before targeting PubMed. Validation **flags, never blocks**: an invalid term is saved-and-marked for correction, not rejected. The bundled MeSH artifact's **presence/integrity MUST be verified at startup**; if it is missing or unreadable, MeSH terms MUST be marked **`unvalidated`** (a third state distinct from valid/invalid) and ingestion MUST still proceed on the watchlist's drugs/keywords — a missing vocabulary artifact MUST NOT crash validation or ingestion.
- **FR-010**: System MUST target the **PubMed** source using the validated MeSH terms and MUST rely on **PubMed's native server-side MeSH expansion** for synonym coverage (the system MUST NOT build its own synonym-expansion engine). MeSH-based targeting applies only to the MeSH-aware sources (PubMed; Europe PMC where supported); openFDA and the regulatory feeds use drug name + keywords with no automatic synonym expansion. A watchlist with no MeSH terms MUST still ingest using its drugs and keywords without error.
- **FR-011**: System MUST treat an ingestion run as a **tracked unit of work** that records, per run: the target watchlist, the acting admin, start/finish times, an **overall status from the enumerated set** `{running, success, partial_success, failed}` (`success` = all sources succeeded; `partial_success` = at least one source succeeded and at least one failed; `failed` = all sources failed or the run could not start), per-source outcomes, and document counts (fetched, created, skipped-as-duplicate, errored), all retrievable scoped to the client.
- **FR-012**: System MUST **isolate per-source failures**: if one source is unreachable, times out, is rate-limited, returns malformed data, or lacks a required credential, the run MUST still ingest from the remaining sources, record the failed source (with its error captured) without aborting the run, and reflect partial success in the overall status.
- **FR-013**: System MUST make every external source call **resiliently**: with a timeout and bounded, backed-off retries for transient failures, while never retrying permanent (e.g., client-error) responses, and MUST respect each source's usage limits (bounded concurrency / polite backoff) so Pantera is not throttled or blocked.
- **FR-014**: System MUST skip and record (as errored) any individual source record that cannot be normalized, without failing the rest of that source's records or the overall run.
- **FR-015**: System MUST treat a source returning zero matching records as a **successful empty result**, not an error.
- **FR-016**: System MUST record every ingestion-run trigger to the existing audit log, attributed to the acting admin via the spec-2 human-actor mechanism and naming the target watchlist (one entry per trigger).
- **FR-017**: System MUST store any external-source credentials (e.g., API keys) only in the secrets store (Vault) via the established path; no source credential may live in source files or environment files, and a missing credential MUST degrade to a recorded failed source rather than a crash. Any **new required secret** this spec adds to `_REQUIRED_SECRETS` MUST also be added to the CI inline secret writer (`.github/workflows/ci.yml`) so migrations/tests do not fail fast (per the spec-2 lesson).
- **FR-018**: System MUST NOT, in this spec, parse documents into chunks, embed them, classify them, or draft reports; this spec's responsibility ends at fetching, normalizing, tagging, deduplicating, and persisting documents (and validating MeSH terms). Downstream processing is owned by later specs.
- **FR-019**: New schema introduced by this spec MUST follow the established database conventions (carry `client_id`, include the required indexes — at minimum on `client_id` and on the external-identifier dedup key) and MUST be delivered as a new versioned migration that does not break the spec-1/2/3 schema.
- **FR-020**: System MUST validate all trigger inputs at the API boundary and MUST reject malformed input with clear, non-leaking validation messages, leaving persisted state unchanged on rejection.
- **FR-021**: System MUST ingest **incrementally**: each run MUST fetch only records newer than a **last-successful-run watermark** tracked per `(watchlist, source)`, and MUST advance a source's watermark only when that source's portion of the run succeeds (a failed source re-attempts its window on the next run). "Newer" MUST be measured against a defined record attribute — the record's **publication/alert date**, falling back to the source's **indexed/added date** where no publication date exists; a source that exposes **no usable date** MUST instead page by its native ordering/cursor and rely on dedup (FR-006) to suppress already-seen records. The **first** run for a `(watchlist, source)` MUST use a bounded, configurable **initial lookback window** (default ~12 months), and every source query MUST be bounded by a configurable **per-source result cap** as a safety valve against unbounded result sets. Dedup (FR-006) remains authoritative against any overlap the windowing produces.
- **FR-022**: System MUST define the lifecycle of ingested data when its watchlist or client becomes inactive: deactivating a watchlist or suspending a client MUST stop new ingestion for it (no new runs accepted) while **preserving** already-ingested documents, provenance links, run records, and watermarks (no destructive delete in this spec), so the data and audit trail remain intact. Hard deletion / right-to-erasure of this data is a later-spec concern and out of scope here.
- **FR-023**: System MUST persist ingested source records (including openFDA FAERS payloads, which contain de-identified patient attributes such as age/sex/country but no direct identifiers) as the retained raw payload for later parsing, and MUST NOT write patient attributes or secrets into logs or traces. Active PII/secret **redaction** (Presidio) before downstream parsing/embedding/use is owned by spec 12, which interposes before the corpus is consumed; this spec's obligation is faithful storage plus PII-free logging.
- **FR-024**: System MUST keep an ingestion run's recorded status truthful even if execution is interrupted: a run left in `running` because the process stopped mid-execution (the in-process background model has no cross-restart durability) MUST be reconcilable to a terminal `failed` state (e.g., via a startup sweep or staleness timeout) so no run is reported as perpetually `running`. Because dedup (FR-006) and per-`(watchlist, source)` watermarks make re-triggering safe and idempotent, a re-run after such a failure MUST NOT create duplicates. (Durable queued execution arrives with ARQ in spec 11.)

### Key Entities *(include if feature involves data)*

- **Document**: A single ingested publication/record, normalized to the common shape, representing **one real paper/record per client** (deduplicated across sources). **Client-scoped** (carries `client_id`, not a single owning watchlist). Key attributes: owning `client_id`, **normalized external identifier** (the dedup key), the set of **contributing source(s)**, **source-reliability** tier (the highest among contributing sources), title/summary, publication or alert date, origin reference (URL), **fetched-at** timestamp, and the retained **raw source payload(s)**. Unique per client by `(client_id, normalized_external_id)`. Linked to the watchlist(s)/run that surfaced it via a separate **provenance** relationship (a paper relevant to several of a client's watchlists is stored once and linked to each). The corpus this and every later spec build on. (Drug-disambiguation / canonical-drug identity is a documented future improvement, not in this spec.)
- **Document Provenance Link**: The relationship recording which watchlist(s) and ingestion run(s) surfaced a given document for its client — the within-client many-to-many between watchlists and documents that lets a shared paper be stored once while still showing why it was pulled in.
- **Source / Source Adapter**: An external origin of records (PubMed, Europe PMC, openFDA FAERS, openFDA drug labels, FDA MedWatch, EMA, MHRA) accessed behind a uniform adapter that translates the source's native format and query semantics into the common document shape and assigns the source-reliability tier. The set of configured sources is platform-level configuration.
- **Source-Reliability Tier**: An ordered classification of how authoritative a document's origin is — `regulatory_alert` (highest) > `peer_reviewed` > `preprint` > `case_report` — stored on each document and consumed by later corroboration (spec 7) and triage (spec 8).
- **Ingestion Run**: A triggered unit of ingestion work for one watchlist. Attributes: target watchlist, acting admin, start/finish times, overall status from `{running, success, partial_success, failed}`, per-source outcomes, and document counts (fetched / created / skipped-duplicate / errored). A run interrupted mid-execution is reconciled from `running` to `failed` (FR-024). The audited, observable record of an ingestion cycle; the scheduled automation that creates these runs is spec 11.
- **Source Watermark**: The per-`(watchlist, source)` "last successfully ingested up to" marker that makes runs incremental. Advanced only when that source succeeds in a run; absent on the first run, where a configurable initial lookback window applies instead. Bounds how far back each run queries each source.
- **MeSH Term (validated)**: A watchlist's medical subject heading, now validated/resolved against the bundled slim MeSH list (the resolution deferred from spec 3). Carries a **persisted validity state** from `{valid, invalid, unvalidated}` (set at save time, surfaced on every watchlist read; `unvalidated` is used when the bundled MeSH artifact is unavailable per FR-009); valid terms are used to target PubMed (PubMed performs the synonym expansion), invalid terms are flagged for admin correction. This extends the spec-3 `watchlist_items` MeSH entries (free-form `value`/`normalized_value`) with a validity result rather than replacing them.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An admin can trigger an ingestion run for their own active watchlist and have the matching external records persisted as client-scoped documents, each with its source, external identifier, normalized metadata, reliability tier, and fetched-at time — verified end-to-end for at least one source.
- **SC-002**: All configured source types (PubMed, Europe PMC, openFDA FAERS, openFDA drug labels, FDA MedWatch, EMA, MHRA) are ingested through the uniform adapter contract and normalized into the common document shape, with each document carrying the correct source-reliability tier — verified per source type.
- **SC-003**: Re-running ingestion for an unchanged watchlist creates **zero** duplicate documents, and a record surfaced by two sources within one run is stored once — verified by repeated and overlapping runs reporting accurate created-vs-skipped counts.
- **SC-004**: Zero cross-tenant access: no admin or reviewer can trigger ingestion for, read, or enumerate another client's documents or runs — verified across multiple clients with zero incorrect allows.
- **SC-005**: Only admins can trigger runs; every reviewer or unauthenticated trigger attempt is refused, while reviewers can still view their own client's documents and run results — zero incorrect allows or denies.
- **SC-006**: Watchlist MeSH terms are validated against the controlled vocabulary — valid terms are resolved and used (with expansion) for PubMed targeting and invalid terms are flagged with a clear message — verified with a mix of valid and invalid terms; the spec-3 carryover is closed.
- **SC-007**: A run in which one source fails still ingests from all healthy sources, records the failed source as failed without aborting, and reports accurate per-source outcomes and per-status document counts — verified by inducing a source failure.
- **SC-008**: Every ingestion-run trigger produces exactly one audit-log entry attributed to the correct acting admin and naming the target watchlist.
- **SC-009**: The ingestion write code paths meet the constitution's elevated coverage bar (95%+ on database-write paths), and the overall suite stays at or above the 80% gate.
- **SC-010**: Ingestion is incremental — a second run after a first uses the per-`(watchlist, source)` watermark so that, with no new external records, it fetches no records older than the watermark and creates zero documents; the first run honors the bounded initial lookback — verified by comparing first-run vs second-run fetch scope.
- **SC-011**: An ingestion run never remains reported as `running` after the process stops: an interrupted run is reconciled to `failed`, and a subsequent re-run creates no duplicate documents — verified by simulating an interrupted run followed by a re-trigger.

## Assumptions

- **Backend/API only**: This spec delivers the ingestion data model, adapters, dedup, source-reliability tagging, MeSH validation, and the manual-trigger API surface. The Admin Console "manual trigger" button and any document-browsing UI are deferred to a later frontend slice; "manual trigger" in the build plan is satisfied at the API level here.
- **Scope ends at persistence**: This spec fetches, normalizes, tags, deduplicates, and stores documents (plus validates MeSH). Parsing into typed chunks and embedding is **spec 6**; hybrid retrieval/corroboration is **spec 7**; triage/classification is **spec 8**; report drafting is **spec 9**. The retained raw payload exists precisely so spec 6 can parse it later.
- **Manual trigger now, scheduling later**: Ingestion is started on demand by an admin in this spec. The trigger endpoint creates the run record, starts the ingestion as an **in-process background task**, and returns immediately with the run ID; the caller reads/polls the run record for status and counts (the run record, not the HTTP response, is the source of truth for results). The automated cadence (biweekly/cron cycles, ARQ-queued execution, dead-letter handling, idempotency, SLA, cost attribution) is **spec 11** (scheduling). The ingestion routine is written as a reusable, independently-testable function (callable directly in tests) that spec 11 will enqueue to the ARQ worker — so moving from background task to ARQ job is minimal rework, not a rewrite. **Durability caveat**: the in-process background task is not durable across a process restart; FR-024 requires interrupted runs to be reconciled from `running` to `failed`, and dedup + watermarks make re-triggering safe. Durable, queued, retryable execution arrives with ARQ in spec 11.
- **Dedup is per-client by `(client_id, normalized_external_id)`**: Documents are tenant-scoped. The dedup key is a **normalized external identifier** chosen by precedence **DOI → PubMed ID → source-native alert/record ID**. **Across sources**, one real paper is stored **once** even when several sources (e.g., PubMed + Europe PMC) return it — the document records all contributing sources and takes the highest contributing reliability tier (so corroboration never counts one paper as two, and spec 6 never embeds it twice). **Within** a client, a paper surfaced by several of that client's watchlists is stored **once** and linked to each via the provenance relationship. **Across** clients, the same record is intentionally stored once **per client** — no cross-tenant document sharing in this spec — because strict multi-tenant isolation (a constitution non-negotiable) outweighs the storage saving; document payloads are small (metadata + abstract/summary). A `client_documents` junction table that stores a shared paper once across clients (and shared chunk embeddings) is a documented **future improvement**, deliberately deferred until the isolation guarantees are proven.
- **Source-reliability is a fixed enum**: The tiers `{regulatory_alert, peer_reviewed, preprint, case_report}` are a small documented enum established here and reused by specs 7–8; refining the mapping rules (e.g., distinguishing label vs. article) is an implementation detail bounded by this enum. Modeled as `String` + CHECK mirrored by a `StrEnum`, per the spec-3 pattern.
- **Reuse of spec-1/2/3 foundations**: The `client_id` tenant boundary (spec 1), the audit log with the human-actor foreign key and the role guards (`require_admin`, `require_reviewer`, `current_active_user`) (spec 2), and the `clients`/`watchlists`/`watchlist_items` tables (spec 3) are reused; this spec adds the `documents` table, the document↔watchlist provenance link, and an ingestion-run record (all carrying `client_id`) as a new migration. Watchlist items' free-form MeSH `value`/`normalized_value` columns (spec 3) are the input to MeSH validation here, extended with a persisted validity state.
- **MeSH validation touches the spec-3 watchlist write path**: Because validation runs at save time, this spec **enhances** the existing spec-3 watchlist add/edit MeSH path to call the new validator and persist the validity flag — an expected extension, since spec 3 explicitly deferred MeSH validation to spec 4. The change flags (never blocks) invalid terms, preserving spec 3's warn-don't-hard-fail behavior.
- **MeSH validation source**: Validation/resolution is against a **bundled slim MeSH term list** shipped with the application (term → canonical form; a few MB, not the full ~hundreds-of-MB descriptor tree), so validation is offline, deterministic, and CI-testable with no per-validation external call. The list is refreshed periodically (MeSH updates roughly yearly); the refresh procedure is an operational/implementation detail. Validation does not silently mutate the admin's stored terms.
- **Synonym expansion is PubMed's job; per-source query fields differ**: The system does not build a synonym-expansion engine — it relies on PubMed's native server-side MeSH expansion. MeSH applies only to MeSH-aware sources (PubMed; Europe PMC partially); openFDA and the regulatory feeds (MedWatch/EMA/MHRA) are queried by drug name + keywords, so synonym coverage there comes from the keywords the admin lists. Automatic cross-source drug-synonym normalization (RxNorm CUI / drug entity disambiguation) is a documented **future improvement**, out of scope in this spec.
- **Document text depth**: This spec stores normalized metadata plus the raw source payload (e.g., abstract/summary + the source's returned record). Fetching and parsing full-text bodies/PDFs is a spec-6 concern; PDF handling is an explicit stretch goal in the source design.
- **Guardrails/redaction layered later**: Injection scanning of ingested text (NeMo Guardrails) and PII/secret redaction (Presidio) before downstream use are **spec 12**; this spec persists source records faithfully (raw payload retained) so spec 12 can interpose before parsing/embedding. No PII is logged in this spec's structured logs.
- **Resilience discipline**: Every external call uses a timeout and bounded exponential-backoff retries (no retry on permanent client errors) and respects source usage limits; this satisfies the constitution's "retries on all external calls" standard. The precise retry/concurrency parameters are implementation details.
- **Incremental fetch windowing**: Runs are incremental against a per-`(watchlist, source)` last-successful-run watermark; the first run uses a configurable initial lookback window (default ~12 months) and every source query is bounded by a configurable per-source result cap. The window default and cap live in `Settings` (non-secret config); their exact values are tunable and refined when the recurring scheduler (spec 11) is built. Dedup remains the authoritative guard against any windowing overlap.
- **Secrets and config discipline**: External-source API keys (e.g., openFDA) go to Vault via the established path and are loaded at startup; non-secret settings live in `Settings`. If this spec introduces a new required secret, it MUST also be added to the CI secret writer (per the spec-2 lesson) so migrations/tests do not fail fast.
- **Operator vs admin trigger**: Per-client manual triggering is an `admin` action on the admin's own client. A cross-tenant platform-operator trigger surface (and the deferred platform-operator console) remains out of scope here, consistent with the spec-3 deferral.
- **Drug disambiguation deferred**: Canonical drug identity (e.g., RxNorm CUI normalization across "Aspirin"/"ASA"/"acetylsalicylic acid") is a documented future improvement and is **not** part of this spec; documents store the source's own drug references as-is.
